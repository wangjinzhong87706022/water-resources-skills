## 摘要

本发明公开了一种基于大语言模型（LLM）的多Skill协同查询方法及系统，涉及自然语言处理和数据库查询技术领域。该方法包括：以声明式Markdown文件定义各业务领域的Skill知识单元，由LLM读取Skill文件自主完成Skill识别、SQL生成与结果解析；构建执行依赖与数据关联的双图分离策略，通过预定义业务依赖规则的有向无环图推导Skill串并行执行计划，同时以独立的因果关联图驱动跨Skill数据融合；对多Skill查询结果执行三维关联分析（时间重叠度/空间共享度/业务因果强度），自适应推断时间粒度，构建加权融合图并自动选择主融合维度；对融合后数据进行基于领域阈值的冲突检测与多策略消解。本发明在水利监测领域的实验结果表明，62道跨6个业务领域的复杂测试题通过率100%，平均得分0.88，有效回答率100%，数据融合完整性达95%以上。

**关键词**：大语言模型；自然语言转SQL；Skill编排；依赖图；数据融合；冲突消解

## 背景技术

### 背景技术1：基于LLM的NL2SQL技术现状

自然语言转SQL（NL2SQL）技术旨在将用户的自然语言查询转换为可执行的SQL语句。随着大语言模型（LLM）的发展，基于LLM的NL2SQL技术取得了显著进展。

现有方案主要存在以下问题：

**方案一：单一Schema方案**
将数据库所有表结构、字段信息、业务规则统一组织在一个Schema中。当数据库表数量多（>50张表）、业务领域复杂（>5个领域）时，Schema过于庞大，导致LLM推理复杂度急剧上升、准确率下降。在水利监测系统中，实测单Schema方案的查询准确率约为76%。

**方案二：固定工作流方案**
采用预定义的工作流节点按固定流程执行。工作流缺乏灵活性，新增业务场景需重新编排流程，难以适应多领域交叉查询的复杂需求。

**方案三：微服务编排方案**
将每个业务领域封装为独立微服务，通过服务编排协调执行。该方案架构复杂、部署成本高，且服务间通信开销大，不适合中小规模监测系统的快速部署。

### 背景技术2：多领域数据融合的挑战

在水利监测等多领域系统中，用户查询常涉及跨领域数据关联（如"降雨对水位的影响"），面临以下技术挑战：

1. **Schema异构性**：不同业务领域使用不同的表结构、字段命名和数据库实例（如水情库sl323、水质库sl325、预测库slztk），无法通过标准JOIN操作直接关联
2. **监测站点命名不一致**：同一物理站点在不同子系统中名称不同（如"古运河水位站"vs"古运河水质站"），缺乏统一标识
3. **时间粒度不统一**：不同监测类型的数据采样频率不同（水位5分钟/次，水质1天/次），直接拼接会导致时间维度错位
4. **数据冲突**：多源数据对同一物理量的观测值可能存在差异，需要领域知识驱动的冲突检测与消解

### 背景技术3：现有技术的不足

综合分析，现有技术存在以下具体问题：

| 问题编号 | 问题描述 | 影响 |
|---------|---------|------|
| P1 | 单一Schema知识量大，LLM推理准确率低 | 复杂查询准确率约76% |
| P2 | 工作流固定，难以适应新业务场景 | 扩展性差 |
| P3 | 执行策略固定，无法根据Skill依赖关系动态优化 | 无依赖的Skill被强制串行执行 |
| P4 | 跨领域数据简单拼接，无法识别关联关系 | 数据完整性和一致性差 |
| P5 | 异构数据源缺乏统一的空间关联键 | 跨领域数据无法自动对齐 |

## 发明内容

### 技术问题

本发明要解决的技术问题是：在多业务领域系统中，如何利用大语言模型实现高准确率的自然语言查询，并自动完成多领域数据的智能融合，提高查询准确率和数据融合质量。

### 技术方案

为解决上述技术问题，本发明提出一种基于大语言模型的多Skill协同查询方法及系统，包括以下五个核心技术方案：

#### 技术方案1：LLM可读的声明式Skill定义方法

该方法使用Markdown格式定义业务领域知识单元（Skill），使Skill文件同时作为人类可读的文档和LLM可执行的"程序"：

**步骤1.1：定义Skill结构**
每个Skill文件包含以下声明式指令段：
- 元数据（name, description, version, tags）：用于Skill索引和路由
- 适用条件表（When to Use）：定义Skill的触发场景和排除条件
- 前置条件（Prerequisites）：声明依赖的数据库、模块和参考文档
- 工作流程（Workflow）：定义LLM应遵循的查询步骤序列
- 关键表（Key Tables）：列出数据库表名、列名、主键、分区键等元信息
- 业务规则（Business Rules）：声明领域特有的计算规则和判断标准
- 关联Skill（Related Skills）：声明与其他Skill的业务关系

**步骤1.2：Skill作为LLM上下文注入**
当用户发起查询时，系统将匹配到的Skill文件全文注入LLM的上下文窗口，LLM基于Skill中定义的Workflow和Business Rules自主生成SQL、执行查询、解析结果。

**步骤1.3：零代码扩展**
新增业务领域只需新增一个Skill Markdown文件，无需编写代码、无需注册服务、无需训练模型。

#### 技术方案2：执行依赖与数据关联的双图分离策略

该方法将Skill间的关系建模为两个独立的图结构，分别用于执行策略推导和数据融合分析：

**步骤2.1：构建执行依赖图（EDG）**
基于预定义的业务执行依赖规则构建有向无环图。所述执行依赖图仅包含硬性前置条件关系，即"Skill B的执行必须等待Skill A的输出完成"。图的节点为Skill，边为执行依赖关系，边不携带权重。

定义的执行依赖规则：
```
EDG = {
    "rainfall":            → []                      # 降雨查询无前置依赖
    "water-situation":     → []                      # 水位查询无前置依赖
    "water-quality":       → []                      # 水质查询无前置依赖
    "water-forecast":      → []                      # 预测查询无前置依赖
    "gate-pump-operation": → []                      # 闸泵查询无前置依赖
    "water-warning":       → [water-situation]       # 预警依赖水位数据
}
```

**步骤2.2：构建数据关联图（DCG）**
基于预定义的业务因果关联规则构建加权无向图。所述数据关联图描述Skill间数据的因果关联强度，仅用于融合阶段的数据关联分析，不参与执行顺序决策。

定义的数据关联规则：
```
DCG = {
    (rainfall, water-situation):     strength=1.0  # 降雨量变化影响河道水位
    (water-situation, water-warning): strength=1.0  # 水位超警戒触发防洪预警
    (water-quality, water-warning):   strength=1.0  # 水质指标决定水质等级
    (water-situation, water-quality): strength=0.7  # 水位变化可能影响水质（弱）
    (gate-pump, water-situation):     strength=0.9  # 闸门启闭影响上下游水位
    (water-forecast, water-situation):strength=0.9  # 预测水位与实际水位对比
}
```

**步骤2.3：双图分离的协同工作流程**
- 执行阶段：使用EDG推导执行计划，拓扑排序后按并行组依次执行
- 融合阶段：使用DCG识别数据关联，构建融合图，执行智能融合

关键约束：数据关联图中的边不构成执行依赖。例如，降雨(rainfall)和水位(water-situation)之间存在强因果关联(strength=1.0)，但两者是独立的数据源，可并行查询。

#### 技术方案3：基于DAG拓扑排序的自适应执行策略推导方法

**步骤3.1：依赖图构建**
根据步骤2.1的EDG，为请求的Skill集合构建子图。仅保留请求集合内存在的依赖边。

**步骤3.2：Kahn算法拓扑排序**
使用Kahn算法对依赖子图进行拓扑排序。算法自然产生并行分批：
1. 计算各节点入度
2. 入度为0的节点组成当前并行批次
3. 移除当前批次节点，更新后继节点入度
4. 重复直到所有节点处理完毕

**步骤3.3：并行分组与策略分类**
排序结果为有序的并行批次列表，每个批次内的Skill可并行执行。根据批次结构将执行策略分类为：
- single：单个Skill
- parallel：所有Skill在一个批次
- serial：每个批次仅含一个Skill
- hybrid：混合并行与串行

**步骤3.4：共享表分析**
识别被多个Skill共同访问的数据库表，作为潜在的数据融合关联点。

#### 技术方案4：基于三维关联分析的自适应数据融合方法

**步骤4.1：时间关联分析**
对每对Skill结果集合，自动检测时间字段（支持tm/time/datetime/时间等多语言字段名），计算：
- 时间范围重叠区间：overlap = [max(start_i, start_j), min(end_i, end_j)]
- 关联强度：strength_time = overlap_duration / min(duration_i, duration_j)
- 自适应时间粒度推断：根据平均采样间隔自动判定对齐粒度

所述自适应时间粒度推断方法：
```
avg_interval = (duration_i / count_i + duration_j / count_j) / 2
granularity = MINUTE  if avg_interval < 600s
              HOUR    if avg_interval < 7200s
              DAY     if avg_interval < 172800s
              MONTH   otherwise
```

**步骤4.2：空间关联分析**
通过站名归一化建立跨子系统的空间关联键：
- 站名归一化：去除领域特定后缀（水位站/水质站/水文站/雨量站/闸站/泵站）
- 示例："古运河水位站" → "古运河"，"古运河水质站" → "古运河"
- 关联强度：strength_spatial = |shared_normalized_stations| / min(|stations_i|, |stations_j|)

**步骤4.3：业务关联分析**
根据步骤2.2的DCG，检查当前Skill组合中是否存在预定义的因果关联规则。

**步骤4.4：加权融合图构建与策略选择**
- 边权重 = Σ(时间关联强度 + 空间关联强度 + 业务关联强度)
- 主融合维度 = argmax(max_strength_per_dimension)
- 自动选择与主融合维度对应的融合策略（时间对齐/空间对齐/业务逻辑）

**步骤4.5：融合执行**
- 时间对齐策略：按推断的时间粒度分组，合并同时间点数据
- 空间对齐策略：按归一化站名分组，合并同站点数据
- 业务逻辑策略：拼接数据并标注因果关联描述

#### 技术方案5：基于领域阈值的冲突检测与多策略消解方法

**步骤5.1：冲突检测**
对融合后数据按(时间, 站点)复合键分组，在每组内检测同字段的数值差异：
```
conflict = (max(values) - min(values)) > threshold(field)
severity = HIGH  if diff > 3 × threshold
          MEDIUM otherwise
```

所述领域特定阈值表：

| 监测字段 | 冲突阈值 | 单位 |
|---------|---------|------|
| 水位 | 0.05 | m |
| 流量 | 1.0 | m³/s |
| 溶解氧 | 0.5 | mg/L |
| 氨氮 | 0.1 | mg/L |
| 降雨量 | 0.5 | mm |
| 默认 | 0.01 | - |

**步骤5.2：冲突消解**
支持四种消解策略：
- latest：取时序最新值（适用于实时监测数据）
- average：取算术平均值（适用于多源同质数据）
- max/min：取极值（适用于安全监控场景）
- priority：按数据源优先级取值（water-situation > rainfall > water-quality）

### 有益效果

本发明相比现有技术，具有以下有益效果：

**效果1：查询通过率显著提升**
通过声明式Skill定义将领域知识注入LLM，62道跨6个业务领域的复杂测试题（L3级别）通过率从95.2%提升至100%，平均得分0.88。

**效果2：有效回答率达到100%**
所有测试题均产生了包含领域数据的有效回答，有效回答率100%，数值数据率99.2%。

**效果3：多域查询稳定性保障**
消融实验验证，基于领域阈值的冲突检测与消解机制对系统稳定性具有决定性作用。移除该机制后，复杂多域查询的通过率从100%降至75%（2/8题超时失败），平均分下降0.162，验证了该机制在处理多源异构数据时的不可或缺性。

**效果4：智能融合显著提升跨域分析质量**
消融实验验证，三维关联分析+智能融合机制对跨域数据关联分析至关重要。移除该机制后，因果分析类查询（如"降雨对水位的影响"）因无法建立跨域数据关联而超时失败，平均分下降0.100，融合质量下降0.083。

**效果5：执行效率显著提升**
通过双图分离策略，无依赖的Skill（如降雨和水位）被正确识别为可并行执行，避免了因果关联导致的不必要串行化。聚焦消融实验验证，移除双图分离后5域并行查询（防洪调度综合评估）因强制串行执行而超时失败，执行策略测试平均耗时增加57%（304.9s→477.0s）。

**效果5.5：自适应时间粒度保障多采样频率对齐**
自适应时间粒度推断机制根据各数据源的平均采样间隔自动选择对齐粒度（MINUTE/HOUR/DAY/MONTH）。聚焦消融实验验证，移除后需要HOUR级别对齐的查询（如"8-18时逐小时水位对比"）因固定DAY粒度无法提供小时级信息而超时失败，时间粒度测试平均分从1.000降至0.535，粒度准确度从1.000降至0.500。

**效果6：零代码可扩展**
新增业务领域Skill只需添加一个Markdown文件，无需编写代码、无需修改系统架构。

## 具体实施方式

### 实施例1：系统整体架构

本实施例描述系统的整体架构设计。

系统采用基于LLM Agent的四层架构：

**1. 用户交互层**
提供命令行界面（CLI），接收用户的自然语言查询请求，支持单次模式（`hermes -z "问题" -s skill`）和交互模式。

**2. Agent调度层**
包括以下核心组件：
- **Skill路由器**：根据用户问题内容，匹配对应的Skill Markdown文件
- **执行策略引擎（planner）**：基于EDG进行拓扑排序，生成串并行执行计划
- **数据融合引擎（fusion）**：基于DCG进行三维关联分析，执行智能融合
- **数据库助手模块（db）**：提供带超时保护的SQL执行接口

**3. Skill存储层**
以Markdown文件形式存储各业务领域的知识单元。文件系统即 Skill注册中心，新增Skill只需添加文件，无需代码修改或服务注册。

**4. 数据库层**
包括多个MySQL数据库实例：
- sl323：水情、降雨、闸泵工况数据
- sl325：水质监测数据
- slztk：水位预测数据

### 实施例2：声明式Skill定义

本实施例详细描述Skill Markdown文件的定义方法。

以水位查询Skill（water-situation）为例：

```markdown

### 实施例3：双图分离的执行策略推导

本实施例详细描述双图分离策略和执行策略推导算法。

**场景**：用户查询"综合分析降雨对古运河水位的影响"

**步骤1：Skill识别**
系统识别问题涉及两个业务领域：rainfall（降雨）和water-situation（水位）。

**步骤2：查询执行依赖图（EDG）**
```
EDG子图（仅包含rainfall和water-situation）:
  rainfall → []
  water-situation → []
```
两者均无执行依赖，入度均为0。

**步骤3：Kahn拓扑排序**
```
初始: in_degree = {rainfall: 0, water-situation: 0}
批次1: [rainfall, water-situation]  ← 入度均为0，可并行
结果: strategy = "parallel"
```

**步骤4：并行执行查询**
系统同时执行rainfall和water-situation的查询，获取各自数据。

**对比：若使用单一依赖图**
若将因果关联（rainfall→water-situation, strength=1.0）也作为执行依赖，则排序结果为：
```
批次1: [rainfall]        ← 串行
批次2: [water-situation] ← 等待rainfall完成
结果: strategy = "serial"
```
这会导致不必要的串行等待。本发明的双图分离策略避免了这一问题。

**步骤5：查询数据关联图（DCG）用于融合**
```
DCG匹配: (rainfall, water-situation) → strength=1.0, "降雨量变化会影响河道水位"
```
在融合阶段使用此关联进行数据分析。

### 实施例4：三维关联分析与自适应融合

本实施例详细描述数据融合的完整流程。

**场景**：融合rainfall和water-situation两个Skill的查询结果

**输入数据**：
- rainfall结果：10个雨量站，时间范围2026-03-22至2026-03-31，日均数据
- water-situation结果：7个水位站，时间范围2026-03-22至2026-03-31，日均值

**步骤1：时间关联分析**
```
rainfall时间范围: [2026-03-22, 2026-03-31], duration = 9天
water-situation时间范围: [2026-03-22, 2026-03-31], duration = 9天
overlap = [2026-03-22, 2026-03-31], overlap_duration = 9天
strength_time = 9 / min(9, 9) = 1.0

自适应粒度推断:
  rainfall avg_interval = 9天/10条 ≈ 0.9天 ≈ 77760s
  water-situation avg_interval = 9天/10条 ≈ 0.9天
  avg = 77760s → DAY粒度
```

**步骤2：空间关联分析**
```
rainfall站名归一化后: {马港河, 冷却河闸, 二道河, ...} (10站)
water-situation站名归一化后: {古运河(新城河口), 古运河(春江路), ...} (7站)
shared = ∅  (雨量站和水位站不在同一物理位置)
strength_spatial = 0
```

**步骤3：业务关联分析**
```
DCG匹配: (rainfall, water-situation) → strength=1.0
```

**步骤4：融合策略选择**
```
时间关联强度: 1.0
空间关联强度: 0
业务关联强度: 1.0
主维度 = max(time=1.0, spatial=0, business=1.0) → time（并列时优先时间维度）
策略 = 时间对齐（DAY粒度）
```

**步骤5：融合执行**
按DAY粒度分组，将同一天的降雨和水位数据合并：
```
{
  "时间": "2026-03-30",
  "平均降雨量(mm)": 11.4,
  "古运河水位(m)": 4.79,
  "水位较前日变化": +0.08,
  "_sources": ["rainfall", "water-situation"]
}
```

### 实施例5：冲突检测与消解

本实施例详细描述冲突检测与消解方法。

**场景**：融合water-situation和water-warning的数据时，同一站点出现两个水位值

**输入数据**：
```
来源1(water-situation): 古运河(新城河口) 水位=4.78m, 时间=19:00
来源2(water-warning):   古运河(新城河口) 水位=4.82m, 时间=19:05
```

**步骤1：冲突检测**
```
分组键: (时间="2026-03-31", 站点="古运河")
字段"水位"的值: [4.78, 4.82]
差异: |4.82 - 4.78| = 0.04
阈值: 0.05m（水位字段）
判定: 0.04 < 0.05, 未超过阈值, 不构成冲突
```

**步骤2（假设差异超阈值）：冲突消解**
```
若差异为0.12m（超过0.05m阈值）:
  severity = MEDIUM (0.12 < 3×0.05=0.15)
  strategy = "latest"
  resolved = 4.82 (取最新值19:05的数据)
```

### 实施例6：跨域综合查询的端到端流程

本实施例描述一个完整的跨域综合查询流程。

**用户问题**："综合查询古运河水位和降雨情况，分析降雨对水位的影响"

**步骤1：Skill识别**
系统通过water-fusion Skill的关键词映射表识别涉及两个业务域：
- "水位/水情" → water-situation
- "降雨/雨量" → rainfall

```
plan_execution(["rainfall", "water-situation"])

→ 拓扑排序: 批次1 = ["rainfall", "water-situation"]
→ strategy = "parallel"
```

**步骤3：并行查询执行**

water-situation数据: SELECT ... FROM st_river_r JOIN st_stbprp_b WHERE ...
```

correlate({"rainfall": rainfall_rows, "water-situation": water_rows})

detect_conflicts(fused_data)

#### 结果分析

**结论1：冲突检测与消解是系统稳定性的关键保障**

冲突检测与消解在消融实验中贡献最大（+0.162），其核心价值体现在：移除后涉及空间对齐和混合融合的复杂查询（F5: 水位+闸泵沿线查询、F6: 防洪综合分析）均超时失败。原因在于：多源异构数据直接拼接会导致字段冲突和逻辑矛盾，LLM在处理不一致数据时陷入循环推理，最终触发600秒超时。该结果表明，基于领域阈值的冲突检测机制不仅仅是"数据质量控制"功能，更是多Skill协同查询系统能否正常完成复杂查询的稳定性保障。

**结论2：三维关联+智能融合是跨域分析的核心能力**

智能融合机制贡献第二（+0.100），在因果分析场景（F2: "分析降雨对水位的影响"）中体现最为显著：移除后该题从1.00分降至0.10分（超时），因为不做关联分析时LLM无法建立"降雨→水位变化"的因果关系链，查询陷入无方向的反复搜索。

**结论3：双图分离策略保障复杂多域查询的执行成功率**

在初始8题测试中，双图分离策略（+0.000）未体现差异，原因：通用测试集未充分触发串行化瓶颈。为此设计了专项聚焦测试（见实验3），结果表明：(1) 移除双图分离后，5域并行查询（ES4: 防洪调度综合评估）因全部Skill被强制串行执行而超时失败（600s TIMEOUT），得分从1.00降至0.10；(2) 执行策略测试（ES1-ES4）平均分从0.877降至0.652，平均耗时从304.9s增至477.0s（+57%）；(3) 聚焦测试中双图分离贡献+0.113，验证了其核心价值：避免因果关联导致不必要的串行化，在涉及3个以上独立Skill的复杂查询中保障执行效率和成功率。

**结论4：自适应粒度推断对小时级查询至关重要**

在初始8题测试中，自适应时间粒度推断（-0.038）未体现正向贡献，原因：通用测试数据以日级别（DAY）为主，恰好与固定DAY粒度一致。专项聚焦测试（见实验3）揭示了其关键价值：(1) 移除后，需要HOUR级别对齐的查询（TG2: 8-18时逐小时对比、TG3: 降雨过程水位响应）均超时失败（600s TIMEOUT），得分从1.00降至0.07；(2) 时间粒度测试（TG1-TG4）平均分从1.000降至0.535，粒度准确度从1.000降至0.500；(3) 聚焦测试中自适应粒度贡献+0.220，是所有创新点中贡献最大的。该结果表明，自适应时间粒度推断机制在处理不同采样频率数据源时具有不可替代的作用。

## 说明书摘要

本发明公开了一种基于大语言模型的多Skill协同查询方法及系统。以声明式Markdown文件定义各业务领域Skill，由LLM读取Skill文件自主完成查询；构建执行依赖图和数据关联图的双图分离策略，避免因果关联导致不必要的串行化；对多Skill查询结果执行三维关联分析（时间/空间/业务），自适应推断时间粒度，构建加权融合图并自动选择融合策略；基于领域阈值检测并消解数据冲突。62道跨6个业务领域的L3级测试题通过率100%，平均得分0.88。消融实验进一步验证各创新点的独立贡献：冲突检测与消解机制对系统稳定性贡献最大（+0.162，移除后通过率降至75%），自适应时间粒度推断对小时级查询至关重要（+0.220，移除后50%时间粒度测试超时），三维关联+智能融合保障跨域分析能力（+0.100，移除后因果分析查询超时），双图分离策略保障复杂多域查询执行效率（+0.113，移除后5域并行查询超时），四项核心技术协同保障了多域复杂查询的成功率、数据融合质量和执行效率。
