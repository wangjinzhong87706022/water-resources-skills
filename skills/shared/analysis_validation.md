# 水利分析结果验证

> 在向用户/决策者交付分析结果前的质量把关。方法论审查 + 计算验算 + 常见陷阱检查 + 置信度评定。

## 一、审查清单

### 数据质量

- [ ] **来源核实**：使用的表是否符合分析目标？确认是生产库还是测试库
- [ ] **时效性**：数据截止时间是否满足分析需要？标注"数据截至 YYYY-MM-DD HH:mm"
- [ ] **完整性**：时间序列有无意外断点？缺失片段是否已排除并说明原因
- [ ] **空值处理**：关键列的空值率是否已评估？处理方式是排除/插值/标记？
- [ ] **去重确认**：JOIN 是否造成行数膨胀？用 `COUNT(DISTINCT pk)` 验证
- [ ] **过滤正确性**：WHERE 条件是否排除了不应排除的数据？是否漏掉了标准过滤器（test/internal/已撤销站）

### 计算验算

- [ ] **聚合层级**：GROUP BY 列是否覆盖了所有非聚合列？聚合粒度与分析目标是否一致
- [ ] **分母正确性**：率/百分比计算的分母是否合适？分母是否为 0？
- [ ] **日期对齐**：对比周期长度是否一致？不完整周期已排除或标注
- [ ] **JOIN 正确性**：INNER vs LEFT 选择是否合理？多对多 JOIN 是否造成计数膨胀
- [ ] **小计求和**：部分加总是否等于整体？若有差异需解释（如重复计数）
- [ ] **跨库一致性**：sl323/sl325/slztk 的时间方式和单位是否统一？

### 合理性抽查

- [ ] **数值范围**：所有数据在物理合理范围内（水位 -1~20m，降雨 0~500mm，DO 0~20mg/L）
- [ ] **趋势连续性**：时间序列有无突变跳点？是否有合理解释
- [ ] **交叉验证**：关键数字是否与已知数据源一致（如常测站水位是否匹配 SCADA 面板）
- [ ] **边界情况**：空集合/零值/缺失段时查询是否正常运行？
- [ ] **量级验证**：反算验证——"全市总降雨 / 站数 ≈ 单站均值"

### 呈现检查

- [ ] **图表准确性**：柱状图 Y 轴从 0 开始；折线图无截断 Y 轴误导
- [ ] **数字格式**：水位保留 2 位小数、大数用万/亿单位、百分数保留 1 位小数、一致格式
- [ ] **标题准确**：标题讲洞察而非指标名（"超警站增至6个"而非"水位统计"）
- [ ] **可复现**：SQL 和参数已附带，他人能复现
- [ ] **溯源 footer**：末尾附带 provenance（来源表、数据时间、skill 版本）

## 二、常见水利分析陷阱

### JOIN 爆炸

SELECT COUNT(*)
FROM st_river_r
JOIN st_rvfcch_b ON st_river_r.stcd = ...  -- 如果大小写不匹配或关联条件不对

SELECT COUNT(*) FROM st_river_r;             -- 1000 行
SELECT COUNT(*) FROM ... JOIN ...;           -- 3500 行（异常膨胀）

检测：JOIN 后行数如果增加超过主表的 1.2 倍，需检查 JOIN 关系。COUNT(DISTINCT 主表主键) 而非 COUNT(*)。

### 幸存者偏差

"当前全部 588 个水位站运行正常"——但故障站已被从系统剔除，不报送数据。
措施："本结论基于已报送数据的 588 个站，另有 X 站未报数据（可能故障/离线）。"

### 不完整周期比较

"今年7月水位偏低"——但7月才过半，与完整7月对比是错误的。
措施：只比较已完结周期，或注明天数差异。

### 分母漂移

"水质达标率从 85% 提高到 92%"——但采样站网增加了 30 个轻污染河段站。
措施：用固定站点集合做同比，或用基期站网做分母。

### 均值之均值

- A 河段：3 站，平均 DO = 6.5 mg/L
- B 河段：15 站，平均 DO = 4.2 mg/L
- 错误：均值之均值 = (6.5 + 4.2)/2 = 5.35
- 正确：加权平均 = (3×6.5 + 15×4.2) / 18 = 4.58

### 时区/频率不一致

- sl323 水位数据：UTC+8，5 分钟间隔
- slztk 预测数据：UTC+8，1 小时间隔
- 对齐时直接用 tm 关联可能错位，需先统一时间粒度

### 选择偏误

"开展了生态补水的河段水质显著优于未补水河段"——但补水的河段本就是水质相对较好的。
措施：用 DID（双重差分）或 PSM（倾向得分匹配）控制选择偏误，或坦陈此局限。

### 辛普森悖论

每个水质指标都改善，但综合评价恶化——因为采样向量向重污染断面倾斜。

### 时间窗选择

"今年汛期降雨量创 10 年新低"——但汛期定义若从 5/1-9/30 调整为 6/1-9/30，结论可能逆转。
措施：固定时间窗定义，或同时在多个窗宽下验证。

### 测站编码多体系无法直接 JOIN

水位/水文站（`st_river_r`，编码如 `HT0021…`/`5100…`）、闸泵站（`st_was_r`，`HPBZCZ*`）、
水质站（`sl325.wq_pcp_d`）使用**不同编码体系**，跨域按 `stcd` 硬 JOIN 会丢数据或错配。
- 检测：JOIN 后 `COUNT(DISTINCT stcd)` 远小于两表任一方 → 编码体系不匹配。
- 措施：跨域关联走**站名归一化**（见下）或 `lib/fusion.py:normalize_station()` 按名匹配，而非编码硬 JOIN。

### 站名后缀归一化

同一测站在不同子系统名称带不同后缀（"某站水位站"/"某站水文站"/"某站(水质)"），按 `stnm`
直接关联会漏配。
- 检测：两站名集合剥离后缀后交集增大 → 原本因后缀漏配。
- 措施：剥离后缀（水位站/水文站/水质站/雨量站/闸站/泵站）再匹配；`fusion.normalize_station()` 已实现。

### 传感器故障模式 vs 真实极值

水位/水质**突跳**（单点尖刺）、**漂移**（缓变偏离）、**干涸**（水位趋 0 或负）常被误当"真实极值"。
- 区分原则：**单站单点**异常 → 优先怀疑传感器；**多站同步**异常 → 优先当真实事件（如暴雨同步抬升）。
- 措施：突跳用相邻站同时刻交叉验证（空间一致性）；漂移看长期趋势；干涸查该站是否"停报/施工"。

### 汛期/非汛期混合致均值失真

"全年平均水位"把汛期(6-9 月)高水位和非汛期拉平，掩盖季节性。
- 措施：报告水位/降雨/水质时**区分汛期 vs 非汛期**，或用**年同比**(今年 6 月 vs 去年 6 月)消除季节性，
  而非"今年 6 月 vs 今年 5 月"。

## 三、Confidence 评定

### Ready to share（可直接交付）

- 方法论无误、计算已验证、陷阱已排查
- 有改进建议但无阻断项
- 数据时效标注清晰
- 附带 provenance footer

### Share with caveats（附说明交付）

- 大部分正确但有特定局限需向用户说明
- 示例："本分析基于 85% 有数据站（65 站缺报），结论在数据完整站成立"

### Needs revision（需修订后交付）

- 存在计算错误、方法论问题或遗漏关键分析
- 列出必须修复的问题及优先级

## 六、结果异常红旗（出现即应深查）

| 红旗 | 含义 | 水利示例 |
|------|------|---------|
| 关键指标环比/同比变化 >50% 且无明确成因 | 多半是数据问题，不是业务突变 | 某站日均水位环比 +300% → 传感器突跳 |
| 结果恰好是整数/规整值 | 常是默认值/截断/过滤问题 | "总降雨 100mm"、"水位 5.00m" 高度聚集 |
| 比率恰为 0% 或 100% | 多半数据不全（空集/单值） | "达标率 100%" 但只查到 1 个站 |
| 结果完美支持假设 | 现实通常更"脏"，过度吻合要警惕 | 预测与实测逐点吻合 → 可能同源/未真正独立 |
| 跨期/跨段数值完全相同 | 查询可能忽略了某个维度 | 所有站某日水位完全相同 → GROUP BY 漏维度 |
| 行数是规整倍数 | JOIN 膨胀或笛卡尔积 | 结果正好是主表行数的整数倍 |

## 交叉验证技术（不只靠"看起来对"）

1. **同指标两种算法**：均值先用 `AVG()` 算一遍，再用 `SUM/COUNT` 算一遍，比对（5.7/8.0 均可）。
2. **单点溯源**：抽 1-2 个具体测站，手工追它的原始记录是否对得上聚合结果。
3. **对标基准**：与已知来源对比（SCADA 面板、历史报告、上级通报、水文年鉴）。
4. **反算闭合**：总降雨 ÷ 站数 ≈ 单站均值？各区之和 ≈ 全市？对不上即有漏算/重复。
5. **边界探测**：过滤到单日/单站/单类，看微结果是否仍合理（空集、零值、单站场景）。
